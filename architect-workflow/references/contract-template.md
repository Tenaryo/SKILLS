# Contract Document Template

## 文档命名规范

```
contracts/v<version>_<module>_<brief>.md
```

示例：
- `contracts/v1_user_auth_api.md`
- `contracts/v2_payment_module_design.md`
- `contracts/v1_system_architecture.md`

## 文档结构

下面定义了契约文档的完整结构。章节标注 **[必写]** 的必须包含，标注 **[按需]** 的仅在用户明确提及时才写入。

### 7. 错误处理规范 **[按需]**

```markdown
## 7. 错误处理规范

### 7.1 错误分类

| 类别 | 说明 | 处理策略 |
|------|------|---------|
| **参数错误** | 调用者传入无效参数 | 立即返回错误，不重试 |
| **业务错误** | 业务规则不满足（如余额不足） | 返回业务错误码，由调用者决定 |
| **系统错误** | 基础设施故障（如 DB 不可用） | 有限重试，最终返回错误 |
| **恐慌错误** | 不可恢复的内部状态错误 | 记录崩溃信息，终止进程 |

### 7.2 错误传播规则

- 参数错误：调用边界立即校验，不向下传播
- 业务错误：由产生错误的模块返回，逐层向上传播或转换为上层错误码
- 系统错误：每层添加上下文信息后向上传播
```

## 填写指南

### 什么必须精确：接口签名

接口签名是契约文档中最重要的部分。开发者根据它编写实现，测试者根据它编写测试，审查者根据它判断设计合理性。**签名必须精确到可以直接写入头文件**：

**✅ 好的签名定义：**
```cpp
/**
 * 根据用户ID获取用户信息。
 * @return 成功时返回UserInfo，失败时返回ErrorCode。
 *         失败情况：E_NOT_FOUND（用户不存在）、E_TIMEOUT（查询超时）
 */
std::expected<UserInfo, ErrorCode> GetUserInfo(uint64_t user_id);
```

**❌ 不好的签名定义：**
```
GetUserInfo: 根据user_id获取用户信息，返回用户信息或错误
```
（缺少类型、缺少错误码枚举、缺少边界条件）

### 什么必须精确：数据类型

所有共享数据结构必须定义到字段级别，包括：
- 字段类型（精确到语言级别的类型）
- 语义含义
- 约束条件（范围、格式、非空等）

### 什么不必包含

以下内容不属于契约文档的范畴，**不要写入**：
- 实现细节（算法选择、数据结构内部布局、缓存策略）
- 代码逻辑（if/else 分支的具体实现）
- 测试用例（具体的测试输入/输出值）
- 构建配置（CMakeLists、编译选项）
- 部署方案（容器编排、网络拓扑）
